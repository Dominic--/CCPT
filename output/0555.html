First, think about these two lines:

> if ((pids[i] = fork()) == 0) {
> lab6_set_priority(i + 1);


After fork() the new child process is added to run queue, but it's just
RUNNABLE, not RUNNING.
While its priority will be set only when it's RUNNING.
Therefore, after the FOR loop, all child processes have a default priority
(which is 1). The run queue look like this:
{ (7,1) (6,1) (5,1) (4,1) (3,1) }, which pair stands for *(pid,priority)*,
ignoring the structure how they're stored in the heap

Then the code reaches the seconde FOR loop, and the waitpid(pids[i],&status[
i]); will invoke the schedule() function.
Inside that function, sched_class_pick_next() is called.
According to the Doc 6 Page 11, the function should increase the processes'
strides (proc->lab6_stride += BIG_STRIDE / proc->lab6_priority)
But at that time, proc->lab6_priority has not been set yet. And it equals 1.
This will bring in errors when the user *priority.c* runs, and the acc
ratio is not the expected correct answer 1:2:3:4:5.
